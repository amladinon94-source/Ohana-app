Moka — layout and user flows

          
Ohana's layout mode: you build the project's user flow / sitemap before (or while) you build it. Flows belong to the project — they live in the "Flows" section of the left navigator and in .ohana/flow.json as living documentation, readable by you and the agent. Opening a flow takes you straight to Moka; there are no modes to switch.


          
Sitemap ⇄ User flow

          
The type is chosen when you create the flow ("+" button in the Flows section) and Moka arranges it accordingly:

          
    
            
  • Sitemap — navigation architecture: a vertical hierarchy (level 1 → level 2/3). Laid out as a tree.
    • 
            
  • User flow — the sequence of a task: horizontal (step 1 → 2 → 3), with branches at decisions. Laid out in layers (Sugiyama).
    • 
          


          
Node types

          
    
            
  • Screens (full card): Page, Modal, Dialog, Navigation. Each one: status (To build / In progress / Done) with a filter, and APIs + links (several, with their context).
    • 
            
  • Decision (diamond): comes with its outputs already resolved — Yes (green) and No (red). It's not a free-form FigJam: the logic stays explicit.
    • 
            
  • System response: a compact node with Success / Error outputs.
    • 
            
  • Start and End: pills that mark where the experience begins and ends — the export uses them so the agent understands the journey.
    • 
          


          
Anatomy of a screen: Section → Component

          
You design it the way you design a real screen:

          
    
            
  • + Section (at the screen level): you add sections (e.g. "Hero", "Filters"). Each section has an editable name and a ⋯ menu (Delete section · Turn into component). Variants: Content / Empty state.
    • 
            
  • + Component (inside a section): you pick from the base catalog (Accordion, Alert, Avatar, Badge, Breadcrumbs, Button, Button group, Calendar, Carousel, Chart, Checkbox, Collapsible, Search, Menu, Data table, Date picker), from your components (global library with name + icon), or from the project's components (your design system, in repos).
    • 
            
  • For each component you note its detail — title, description (what it says / how it looks / how it works) and elements.
    • 
            
  • Layout by presets: the card can start from a SaaS layout (Sidebar, Topbar, Master-detail…) or from the layout editor (split into columns/rows, save the preset per project).
    • 
          


          
Connections

          
    
            
  • Outputs by side: blue = normal connection · green = positive (Yes/Success) · red = negative (No/Error). The color/semantics stays on the edge so the agent understands the logic.
    • 
            
  • Drag = connect (to the side where you drop); click = create a new screen already connected on the opposite side, without overlapping. Grab the line to reroute it; clicking the line opens the editor (direction, label, color, delete).
    • 
            
  • Multi-flow per tab: each flow belongs to a file/repo (src); the switcher shows only those of the active tab, and you can claim "unassigned" flows.
    • 
          


          
Canvas tools

          
    
            
  • Pointer (V): select/move; dragging on empty space = marquee multi-select. Hand (H / Space): pans.
    • 
            
  • Topbar: Search (/) · Filter by status · Fit all (F) · Reload · Snap · Arrange (auto layout) · Layout options · Export.
    • 
            
  • Floating toolbar (insertion palette): Generate with AI · Add screen · Frame · Note · Section title.
    • 
            
  • Notes (stickies) and section titles on the canvas, both with their ⋯ menu. Align/distribute · snap to grid · minimap.
    • 
            
  • Shortcuts: Ctrl/Cmd+C/V copy/paste · Ctrl/Cmd+Shift+C or Alt+drag duplicates · Ctrl/Cmd+Z/Shift+Z undo/redo · Del delete · Ctrl/Cmd+A select all.
    • 
          


          
From Moka to code — prototype or repository

          
The flow isn't a drawing: it's the brief the agent turns into a real product.

          
    
            
  • Export to the agent (topbar): sends the flow as a structured prompt to the tab's terminal — with Start/End, each screen with its sections and components, the APIs/links, and the branch logic.
    • 
            
  • You refer to screens by their name ("build the Login screen", "add a filters section to the Dashboard"); Ohana gives each node an internal id so the agent resolves without ambiguity which one you mean.
    • 
            
  • To an HTML prototype: in the flow topbar, click "To prototype" — it links the flow to an HTML file in prototipos/ and the agent builds it; then "View prototype" opens it in the same project. Visual editing is unrestricted.
    • 
            
  • To a repository: open the repo as a tab; the agent implements each screen of the flow using the project's real design system components — the mockup indicates which component goes in each section.
    •

ohana_flow_listLists all the project's flows and which one is active.

ohana_flow_add_screenAdds a screen (page/modal/dialog/nav/decision).

          
ohana_flow_update_screenEdits name/type/context/position; replaces apis and links.

          
ohana_flow_delete_screenDeletes a screen and the connections touching it.

          
ohana_flow_add_blockAdds a block to a region (with anchor type + items).

          
ohana_flow_add_regionSplits the screen: adds a nested region (row/column).

          
ohana_flow_set_dirChanges a region's direction (row ⇄ column).

          
ohana_flow_set_layoutApplies a SaaS layout preset (sidebar/topbar/…) to a screen.

          
ohana_flow_connectConnects A→B (direction, label, port side).

          
ohana_flow_disconnectRemoves the A→B connection.

          
ohana_flow_layoutArranges the flow (layered layout, LR/TB). Call it last: don't position by hand.


          
How it connects

          
The agent edits .ohana/flow.json with those tools (or you do, on the canvas). Ohana watches the file and refreshes the canvas live while a flow is open (you'll see the loader while the agent builds). "Export to the agent" sends the flow as a structured prompt to the tab's terminal.

        


        
        

          
Comments & Design

          
The review module: Figma-style comments anchored to the prototype, and a design.md as the design source of truth. Both are readable and editable by the agent.


          
How it works

          
    
            
  • Comments: numbered pins anchored to a prototype element (by its data-ai-id or a selector). Threads with replies, resolve/reopen. They live in .ohana/findings.json.
    • 
            
  • Design: design.md holds the prototype's tokens, principles, voice, patterns, and decisions. You view and edit it in the Design panel; it's what the agent reads before touching the UI to keep things consistent.
    • 
          


          
MCP commands (Comments)

          
ohana_statusWhich prototype is open and a summary of comments.

          
ohana_list_commentsLists comments (filters by status).

          
ohana_create_commentCreates a comment anchored to an element.

          
ohana_replyReplies to a thread (and optionally resolves it).

          
ohana_resolveResolves or reopens a thread.

          
ohana_deleteDeletes a comment.


          
MCP commands (Design)

          
ohana_read_designReads design.md (the design source of truth).

          
ohana_update_designRewrites design.md (appears live in the panel).

          
ohana_init_designCreates design.md from a template if it doesn't exist.


          
How it connects

          
Ohana watches findings.json and design.md. When the agent comments, replies, resolves, or updates the design, the change appears live in the app (pins and Design panel).

        


        
        

          
How it all connects

          
Ohana is agent-native: the whole app can be managed from the terminal. The "CLI" isn't a binary — it's the embedded terminal (one per tab) running your agent, connected to the app via MCP and files in .ohana/.


          
The per-tab terminal

          
    
            
  • Each tab has its own shell. The cwd is the repo root (repo tab) or the file's folder (prototype tab).
    • 
            
  • In prototype tabs, the shell exposes $OHANA_FILE and $OHANA_FILE_NAME so the agent knows exactly which file to work on, even when there are siblings in the same folder.
    • 
          


          
The bridge

          
    
            
  • The MCP server reads ~/.ohana/active.json (which Ohana writes when a tab is activated) to locate the active project's .ohana/.
    • 
            
  • Bridge files in .ohana/: findings.json (comments) · design.md (design) · flow.json (Moka) · commands.json (agent→app actions).
    • 
            
  • Bidirectional: the agent reads the state (via MCP) and drives the app — it writes actions to commands.json that Ohana runs on the preview (click, type, assert, screenshot, run a step sequence).
    • 
          


          
  Terminal (per tab)            Ohana (app)
  ┌───────────────┐            ┌──────────────────┐
  │  Your agent   │── MCP ────▶│  tools           │
  │   ($OHANA_FILE)│◀───────────│  ohana_*         │
  └──────┬────────┘            └────────┬─────────┘
         │  reads / writes              │ watches
         ▼                               ▼
     .ohana/  findings.json · design.md · flow.json · commands.json
         │                               │
         └──────── live changes ─────────┘
            (pins, Design panel, Moka canvas, actions on the preview)
