Moka — maquetación y user-flows

          
El modo de maquetación de Ohana: construyes el user-flow / sitemap del proyecto antes (o mientras) lo construyes. Los flujos son del proyecto — viven en la sección «Flujos» del navegador izquierdo y en .ohana/flow.json como documentación viva, legible por ti y por el agente. Abrir un flujo te lleva directo a Moka; no hay modos que cambiar.


          
Sitemap ⇄ User flow

          
El tipo se elige al crear el flujo (botón «+» de la sección Flujos) y Moka lo ordena en consecuencia:

          
    
            
  • Sitemap — arquitectura de navegación: una jerarquía vertical (nivel 1 → nivel 2/3). Se acomoda como árbol.
    • 
            
  • User flow — secuencia de una tarea: horizontal (paso 1 → 2 → 3), con ramas en las decisiones. Se acomoda por capas (Sugiyama).
    • 
          


          
Tipos de nodo

          
    
            
  • Pantallas (tarjeta completa): Página, Modal, Diálogo, Navegación. Cada una: estado (Por construir / En progreso / Listo) con filtro, y APIs + vínculos (varios, con su contexto).
    • 
            
  • Decisión (rombo): trae sus salidas ya resueltas — Sí (verde) y No (rojo). No es un FigJam libre: la lógica queda explícita.
    • 
            
  • Respuesta del sistema: nodo compacto con salidas Éxito / Error.
    • 
            
  • Inicio y Fin: píldoras que marcan dónde empieza y termina la experiencia — el export las usa para que el agente entienda el recorrido.
    • 
          


          
Anatomía de una pantalla: Sección → Componente

          
Se diseña como se diseña una pantalla de verdad:

          
    
            
  • + Sección (a nivel de pantalla): agregas secciones (ej. «Hero», «Filtros»). Cada sección tiene nombre editable y un menú ⋯ (Eliminar sección · Volver componente). Variantes: Contenido / Empty state.
    • 
            
  • + Componente (dentro de una sección): eliges del catálogo base (Acordeón, Alerta, Avatar, Badge, Breadcrumbs, Button, Button group, Calendar, Carrusel, Chart, Checkbox, Colapsable, Buscador, Menú, Data table, Date picker), de tus componentes (biblioteca global con nombre + ícono) o de los componentes del proyecto (tu design system, en repos).
    • 
            
  • A cada componente le anotas su detalle — título, descripción (qué dice / cómo se ve / cómo funciona) y elementos.
    • 
            
  • Layout por presets: la tarjeta puede partir de un layout SaaS (Sidebar, Topbar, Master-detail…) o del editor de layout (divides en columnas/filas, guardas el preset por proyecto).
    • 
          


          
Conexiones

          
    
            
  • Salidas por lado: azul = conexión normal · verde = positiva (Sí/Éxito) · roja = negativa (No/Error). El color/semántica queda en el edge para que el agente entienda la lógica.
    • 
            
  • Arrastrar = conectar (al lado donde sueltas); clic = crea una pantalla nueva ya conectada por el lado opuesto, sin encimarse. Agarra la línea para re-rutearla; clic en la línea abre el editor (dirección, etiqueta, color, eliminar).
    • 
            
  • Multi-flujo por tab: cada flujo pertenece a un archivo/repo (src); el selector muestra solo los del tab activo, y puedes reclamar flujos «sin asignar».
    • 
          


          
Herramientas del lienzo

          
    
            
  • Pointer (V): selecciona/mueve; arrastrar en vacío = marquee múltiple. Manito (H / Espacio): panea.
    • 
            
  • Topbar: Buscar (/) · Filtro por estado · Ver todo (F) · Recargar · Snap · Ordenar (layout automático) · Opciones de orden · Exportar.
    • 
            
  • Toolbar flotante (paleta de inserción): Generar con IA · Agregar pantalla · Frame · Nota · Título de sección.
    • 
            
  • Notas (stickies) y títulos de sección en el canvas, ambos con su menú ⋯. Alinear/distribuir · snap a grilla · minimapa.
    • 
            
  • Atajos: Ctrl/Cmd+C/V copiar/pegar · Ctrl/Cmd+Shift+C o Alt+arrastrar duplica · Ctrl/Cmd+Z/Shift+Z deshacer/rehacer · Supr borrar · Ctrl/Cmd+A todo.
    • 
          


          
De Moka a código — prototipo o repositorio

          
El flujo no es un dibujo: es el brief que el agente convierte en producto real.

          
    
            
  • Exportar al agente (topbar): manda el flujo como prompt estructurado a la terminal del tab — con Inicio/Fin, cada pantalla con sus secciones y componentes, las APIs/vínculos y la lógica de las ramas.
    • 
            
  • Te refieres a las pantallas por su nombre («construye la pantalla de Login», «agrega una sección de filtros al Dashboard»); Ohana le da a cada nodo un id interno para que el agente resuelva sin ambigüedad a cuál te refieres.
    • 
            
  • A un prototipo HTML: en el topbar del flujo, pulsa «A prototipo» — enlaza el flujo a un HTML en prototipos/ y el agente lo construye; luego «Ver prototipo» lo abre en el mismo proyecto. La edición visual es libre.
    • 
            
  • A un repositorio: abre el repo como tab; el agente implementa cada pantalla del flujo usando los componentes reales del design system del proyecto — la maqueta indica qué componente va en cada sección.
    • 
          


          
Comandos MCP (Moka)

          
ohana_flow_readLee el flujo activo: pantallas, edges y globals.

          
ohana_flow_listLista todos los flujos del proyecto y cuál está activo.

          
ohana_flow_newCrea un flujo nuevo (y lo activa).

          
ohana_flow_add_screenAgrega una pantalla (page/modal/dialog/nav/decision).

          
ohana_flow_update_screenEdita nombre/tipo/contexto/posición; reemplaza apis y links.

          
ohana_flow_delete_screenBorra una pantalla y las conexiones que la tocan.

          
ohana_flow_add_blockAgrega un bloque a una región (con tipo ancla + items).

          
ohana_flow_add_regionDivide la pantalla: agrega una región anidada (fila/columna).

          
ohana_flow_set_dirCambia la dirección de una región (fila ⇄ columna).

          
ohana_flow_set_layoutAplica un preset de layout SaaS (sidebar/topbar/…) a una pantalla.

          
ohana_flow_connectConecta A→B (dirección, etiqueta, lado del puerto).

          
ohana_flow_disconnectQuita la conexión A→B.

          
ohana_flow_layoutOrdena el flujo (layout por capas, LR/TB). Llámalo al final: no posiciones a mano.


          
Cómo se conecta

          
El agente edita .ohana/flow.json con esas herramientas (o tú en el lienzo). Ohana vigila el archivo y refresca el canvas en vivo con un flujo abierto (verás el loader mientras el agente construye). «Exportar al agente» manda el flujo como prompt estructurado a la terminal del tab.

        


        
        

          
Comentarios & Diseño

          
El módulo de revisión: comentarios estilo Figma anclados al prototipo, y un design.md como fuente de verdad del diseño. Ambos son legibles y editables por el agente.


          
Cómo funciona

          
    
            
  • Comentarios: pins numerados anclados a un elemento del prototipo (por su data-ai-id o un selector). Hilos con respuestas, resolver/reabrir. Viven en .ohana/findings.json.
    • 
            
  • Diseño: design.md guarda tokens, principios, voz, patrones y decisiones del prototipo. Se ve y edita en el panel Design; es lo que el agente lee antes de tocar la UI para mantener consistencia.
    • 
          


          
Comandos MCP (Comentarios)

          
ohana_statusQué prototipo está abierto y resumen de comentarios.

          
ohana_list_commentsLista comentarios (filtra por estado).

          
ohana_create_commentCrea un comentario anclado a un elemento.

          
ohana_replyResponde un hilo (y opcionalmente lo resuelve).

          
ohana_resolveResuelve o reabre un hilo.

          
ohana_deleteElimina un comentario.


          
Comandos MCP (Diseño)

          
ohana_read_designLee design.md (fuente de verdad del diseño).

          
ohana_update_designReescribe design.md (aparece en vivo en el panel).

          
ohana_init_designCrea design.md desde una plantilla si no existe.


          
Cómo se conecta

          
Ohana vigila findings.json y design.md. Cuando el agente comenta, responde, resuelve o actualiza el diseño, el cambio aparece en vivo en la app (pins y panel Design).

        


        
        

          
Cómo se conecta todo

          
Ohana es agente-nativa: toda la app se puede gestionar desde la terminal. El «CLI» no es un binario — es la terminal embebida (una por tab) corriendo tu agente, conectada a la app por MCP y por archivos en .ohana/.


          
La terminal por tab

          
    
            
  • Cada tab tiene su propio shell. El cwd es la raíz del repo (tab de repo) o la carpeta del archivo (tab de prototipo).
    • 
            
  • En tabs de prototipo, el shell expone $OHANA_FILE y $OHANA_FILE_NAME para que el agente sepa exactamente sobre qué archivo trabajar, aunque haya hermanos en la misma carpeta.
    • 
          


          
El puente

          
    
            
  • El servidor MCP lee ~/.ohana/active.json (que Ohana escribe al activar un tab) para ubicar el .ohana/ del proyecto activo.
    • 
            
  • Archivos puente en .ohana/: findings.json (comentarios) · design.md (diseño) · flow.json (Moka) · commands.json (acciones agente→app).
    • 
            
  • Bidireccional: el agente lee el estado (vía MCP) y maneja la app — escribe en commands.json acciones que Ohana ejecuta sobre el preview (click, type, assert, screenshot, ejecutar un flujo de pasos).
    • 
          


          
  Terminal (por tab)            Ohana (app)
  ┌───────────────┐            ┌──────────────────┐
  │   Tu agente   │── MCP ────▶│  herramientas    │
  │   ($OHANA_FILE)│◀───────────│  ohana_*         │
  └──────┬────────┘            └────────┬─────────┘
         │  lee / escribe                │ vigila (watch)
         ▼                               ▼
     .ohana/  findings.json · design.md · flow.json · commands.json
         │                               │
         └──────── cambios en vivo ──────┘
            (pins, panel Design, canvas Moka, acciones sobre el preview)
